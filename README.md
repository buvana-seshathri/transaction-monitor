# transaction-monitor

A real-time transaction anomaly detection pipeline: synthetic transactions
stream through Kafka, get scored by an Isolation Forest model, land in
SQLite, and show up live on a simple dashboard.

Built to mirror the shape of real fraud/risk systems (Kafka-based
streaming + ML scoring) while staying small enough to read end to end —
18 files, no framework bloat, nothing you can't explain in an interview.

## Architecture

```text
transaction_generator.py --> Kafka topic --> consumer.py --> SQLite --> api.py --> dashboard
                (producer.py)          (scores with model.py)
```

Five jobs, five layers: **generate** → **judge** → **store** → **serve** → **display**.

## Project Structure

```text
.
├── transaction_generator.py      # Generates synthetic transactions (normal + anomalous)
├── model.py                      # Isolation Forest wrapper: train / save / load / score
├── train_model.py                # One-time script: builds historical data, trains, saves model
├── historical_transactions.csv   # Historical dataset (generated by train_model.py)
├── model.pkl / encoder.pkl       # Trained model + category encoder (generated by train_model.py)
├── database.py                   # SQLite storage: insert + query scored transactions
├── config.py                     # Shared settings (Kafka broker, topic, timing)
├── producer.py                   # Streams fake transactions onto the Kafka topic
├── consumer.py                   # Reads the stream, scores each transaction, saves it
├── docker-compose.yml            # Single-node Kafka broker (KRaft mode, no Zookeeper)
├── api.py                        # FastAPI: /transactions, /anomalies, /stats
├── index.html / app.js / style.css   # Dashboard: polls the API, renders a live table
├── requirements.txt
└── .gitignore
```

## Isolation Forest

Isolation Forest is an **unsupervised anomaly detection** algorithm that
identifies unusual transactions by randomly isolating data points.

- Builds multiple random decision trees.
- Randomly selects a feature and a split value at each node.
- Anomalous transactions are isolated in **fewer splits** because they are rare and differ from normal data.
- Normal transactions require **more splits** since they are surrounded by similar observations.

The model predicts:
- `1` → Normal transaction
- `-1` → Anomalous transaction

Features used: `amount`, `hour` (extracted from the timestamp), and
`merchant_category` (one-hot encoded).

## Setup

```bash
pip install -r requirements.txt
```

Kafka requires Docker Desktop — install from
[docker.com/products/docker-desktop](https://www.docker.com/products/docker-desktop/)
if you don't already have it.

## Running the full pipeline

### 1. Train the model (one-time)

```bash
python3 train_model.py
```

This generates `historical_transactions.csv`, trains the Isolation Forest,
and saves `model.pkl` + `encoder.pkl`. It also prints a quick sanity check
comparing the model's predictions against the seeded ground-truth labels.

Re-run this any time you want to retrain (e.g. after changing
`transaction_generator.py`'s anomaly logic).

### 2. Start Kafka

```bash
docker compose up -d
```

Confirm it's running with `docker ps` — you should see a container named `kafka`.

### 3. Start the consumer (Terminal 1)

```bash
python3 consumer.py
```

Loads the trained model and waits for messages on the `transactions` topic.

### 4. Start the producer (Terminal 2)

```bash
python3 producer.py
```

Streams one fake transaction per second onto Kafka. Watch Terminal 1 —
you'll see each one scored as `[normal]` or `[ANOMALY]` in real time.

### 5. Start the API (Terminal 3)

```bash
uvicorn api:app --reload --port 8000
```

Visit `http://localhost:8000/docs` for an interactive view of the endpoints:
- `GET /transactions?limit=50` — recent feed, newest first
- `GET /anomalies?limit=50` — flagged only
- `GET /stats` — total count, anomaly count, anomaly rate

### 6. Open the dashboard

Open `index.html` directly in your browser, or serve it locally:

```bash
python3 -m http.server 5500
```

Then visit `http://localhost:5500`. The table polls the API every 3 seconds —
green rows are normal, red rows are flagged anomalies.

### Shutting down

```bash
docker compose down
```

## Design notes

- **Why Kafka in KRaft mode (no Zookeeper):** modern Kafka doesn't need a
  separate Zookeeper container anymore, so the whole broker is a single
  service in `docker-compose.yml` — one less moving part to explain.
- **Why synthetic data instead of a real dataset:** real-world fraud
  datasets (e.g. Kaggle's Credit Card Fraud dataset) anonymize most
  features via PCA, which makes flagged transactions uninterpretable
  (`V14 = -8.2` means nothing to a reader). Synthetic data keeps every
  flag human-readable — `$3,900 at 3am in electronics` — while still
  being scored by a real unsupervised ML algorithm.
- **Why `rowid` instead of `timestamp` for "recent":** transaction
  timestamps are randomized within the last day (to simulate a realistic
  spread of data), so they don't reflect actual arrival order. `database.py`
  sorts by SQLite's built-in insertion-order column instead, so the
  dashboard reflects what actually just streamed in.
- **Dashboard styling:** dark "SOC" (Security Operations Center) theme -
  monospace font, glowing stat cards, subtle CRT scanline overlay, and a
  one-time pulse animation the moment a new anomaly is first seen (tracked
  client-side in `app.js` so it doesn't re-flash on every poll). The table
  view is kept intact underneath the stat cards - full detail, not just
  visual flair.
